# Database URLs, Open Ports, Proxies, and Why My PostgreSQL Connection Did Not Work Like a Website

## The question that caused the confusion

I had a gap in my understanding around this:

> If services like ElephantSQL give me a database URI, why would it be a bad idea to open a port on my own VPS and let my team connect to the database using a URL?

At first, it felt like the same thing.

ElephantSQL gives something like:

```txt
postgresql://user:password@some-host.elephantsql.com/database
```

So I thought I could expose my own database in a similar way:

```txt
postgresql://user:password@my-domain/database
```

The confusing part was that my VPS and Coolify already allow access through URLs like:

```txt
https://coolify.dvloper-z.com
```

but not necessarily through direct IP and port combinations like:

```txt
87.xxx.xxx.xxx:8000
```

So the question became:

> If my apps work through URLs and HTTPS, why can’t my database work the same way?

The answer is: **because a database URL and a website URL are not the same kind of connection.**

---

## The core lesson

A URL or URI is just an address format.

It does not automatically mean:

- the service is public
- the service is secure
- the service goes through HTTPS
- the service goes through the same proxy as a website
- the firewall allows the connection
- the port is open
- the protocol is the same

This was the mental model gap.

I was thinking:

```txt
URL = accessible like a website
```

But the correct model is:

```txt
URL/URI = instructions for how a client should connect to a service
```

Different services use different protocols and ports.

---

## Website URL vs database URI

A regular website uses HTTP or HTTPS.

Example:

```txt
https://coolify.detucel.com
```

This usually means:

```txt
Protocol: HTTPS
Host: coolify.detucel.com
Port: 443
```

If no port is written, HTTPS assumes port `443`.

HTTP assumes port `80`.

So these two are conceptually similar:

```txt
https://coolify.detucel.com
https://coolify.detucel.com:443
```

A PostgreSQL database URI is different.

Example:

```txt
postgresql://user:password@detucel.com:5432/database
```

This means:

```txt
Protocol: PostgreSQL
Host: detucel.com
Port: 5432
Database: database
User: user
Password: password
```

The important detail:

> PostgreSQL does not use HTTPS by default. It uses the PostgreSQL wire protocol, usually over port `5432`.

So even if the host is written as a domain name, the connection is not the same as visiting a website.

---

## The wrong assumption I had

I was mixing these two ideas:

```txt
https://dvloper-z.com
```

and:

```txt
postgresql://user:password@dvloper-z.com/database
```

They both contain a domain, but they are completely different kinds of connections.

The first one goes to a web server or reverse proxy using HTTPS.

The second one tries to connect to a PostgreSQL server using the PostgreSQL protocol.

A reverse proxy like Coolify, Traefik, Nginx, or Caddy commonly handles HTTP and HTTPS traffic.

That means it is usually designed for:

```txt
http://...
https://...
```

Not automatically for:

```txt
postgresql://...
mysql://...
redis://...
```

Some proxies can handle raw TCP traffic, but that requires explicit configuration. It does not happen magically.

---

## Why `coolify.dvloper-z.com` works but `IP:8000` may not

On a VPS with Coolify or similar infrastructure, web traffic often works like this:

```txt
Browser
  ↓
https://coolify.detucel.com
  ↓
DNS points domain to VPS IP
  ↓
Firewall allows 443
  ↓
Reverse proxy receives HTTPS request
  ↓
Proxy routes request based on domain name
  ↓
Coolify app receives request internally
```

This is why the domain works.

But direct access like this may fail:

```txt
87.xxx.xxx.xxx:8000
```

because:

- the firewall may block port `8000`
    
- the app may not be listening publicly on `8000`
    
- Coolify may expect traffic through the proxy
    
- the proxy may route by domain name, not raw IP
    
- HTTPS certificates are issued for domains, not random IPs
    
- only ports `80`, `443`, and `22` may be open
    

That is normal and usually good.

It means the server is not exposing every internal service directly to the internet.

---

## Why opening the database port is risky

PostgreSQL commonly listens on port:

```txt
5432
```

If I open that port publicly, I am allowing external machines to attempt database connections.

That does not mean they can automatically log in, but it does mean they can reach the database service and try.

That creates risks:

- brute-force login attempts
    
- credential leaks becoming catastrophic
    
- misconfigured users having too much access
    
- bots constantly scanning the server
    
- accidental exposure of production data
    
- database denial-of-service attempts
    
- PostgreSQL version or configuration fingerprinting
    

Opening a database publicly is not automatically “wrong,” but it is a serious infrastructure decision.

The safe default is:

```txt
Do not expose the database directly to the public internet.
```

---

## Why ElephantSQL can give me a URI safely

ElephantSQL, Neon, Supabase, Railway, Render, AWS RDS, and other managed database providers can give public database URIs because they are built to do that.

They usually handle:

- firewalling
    
- SSL/TLS configuration
    
- managed users
    
- connection limits
    
- backups
    
- monitoring
    
- high availability
    
- patching
    
- abuse protection
    
- network isolation
    
- secure defaults
    

So when ElephantSQL gives a URI like:

```txt
postgresql://user:password@host.elephantsql.com/database
```

that does not mean “public database URIs are always safe.”

It means:

> A managed database provider has created a controlled public entry point for the database.

If I expose PostgreSQL from my own VPS, I am taking responsibility for what the managed provider normally handles.

In other words:

> Using ElephantSQL does not prove that exposing my own VPS database is safe. It proves that managed providers are useful because they do the hard security work for me.

---

## The key distinction: URI vs exposure

A database URI is just connection information.

Example:

```txt
postgresql://user:password@host:5432/database
```

But the URI does not answer these questions:

- Is the port open?
    
- Is the database listening on the public interface?
    
- Does the firewall allow traffic?
    
- Does the database require SSL?
    
- Is the user restricted?
    
- Is the source IP allowed?
    
- Is this production data?
    
- Is there monitoring?
    
- Are backups configured?
    
- Are credentials rotated?
    

So the URI is only one piece.

The real security question is:

```txt
Who can reach the database service over the network?
```

That is called network exposure.

---

## What happens if I use `postgresql://user:password@detucel.com/database`?

If I omit the port, PostgreSQL clients usually assume port:

```txt
5432
```

So this:

```txt
postgresql://user:password@detucel.com/database
```

is basically treated like:

```txt
postgresql://user:password@detucel.com:5432/database
```

That means the client will try to connect to `detucel.com` on port `5432`.

If the firewall does not allow `5432`, the connection fails.

If PostgreSQL is not listening on the public network interface, the connection fails.

If DNS points to the server but the proxy only handles HTTP/HTTPS, the connection still fails.

A domain name does not automatically route all kinds of traffic.

---

## Why HTTPS does not solve PostgreSQL access

Coolify may use HTTPS certificates for apps.

That is great for web traffic.

But PostgreSQL is not accessed through normal HTTPS.

This works for a website:

```txt
https://app.detucel.com
```

This does not mean the database can be reached through:

```txt
https://db.detucel.com
```

unless I intentionally build some HTTP API in front of the database.

Normally, apps connect directly to PostgreSQL using the PostgreSQL protocol.

So the secure web certificate for the app does not automatically protect raw database traffic.

PostgreSQL can use SSL/TLS, but that is configured separately from website HTTPS certificates.

## Best practice mental model

A production setup usually looks like this:

```txt
Internet
  ↓
HTTPS 443
  ↓
Reverse proxy
  ↓
Application container
  ↓
Private internal network
  ↓
Database
```

The public internet should reach the application, not the database.

The application talks to the database internally.

That means public users interact with:

```txt
https://app.detucel.com
```

but the app internally uses something like:

```txt
postgresql://user:password@postgres:5432/database
```

or:

```txt
postgresql://user:password@private-db-host:5432/database
```

The database host may not even be accessible from outside the server or private network.

That is a good thing.

## Better options for developer access

Sometimes a developer or team member needs to connect to the database from their laptop.

Instead of opening PostgreSQL publicly, better options are:

## Option 1: SSH tunnel

This is one of the simplest and safest options.

The database stays private.

The developer connects through SSH.

Example:

```bash
ssh -L 5433:localhost:5432 user@your-vps-ip
```

Then, on the developer’s machine, they connect to:

```txt
postgresql://db_user:db_password@localhost:5433/database_name
```

What happens here:

```txt
Developer laptop localhost:5433
  ↓
encrypted SSH tunnel
  ↓
VPS localhost:5432
  ↓
PostgreSQL
```

The only public port needed is SSH:

```txt
22
```

This is usually much safer than exposing PostgreSQL directly.

---

## Option 2: VPN / private network

Another good option is to use a private network tool like:

- Tailscale
    
- WireGuard
    
- ZeroTier
    

Then the database only accepts connections from the private network.

Example:

```txt
postgresql://user:password@private-vpn-ip:5432/database
```

This is cleaner when multiple people need access regularly.

---

## Option 3: Managed PostgreSQL

If external database access is required and should be simple, use a managed database provider.

Examples:

- Neon
    
- Supabase
    
- Railway PostgreSQL
    
- Render PostgreSQL
    
- AWS RDS
    
- DigitalOcean Managed PostgreSQL
    

Then the provider gives a URI and handles much of the security/infrastructure.

This is closer to the ElephantSQL model.

---

## Option 4: IP allowlist

If opening the PostgreSQL port is absolutely necessary, do not open it to the whole internet.

Bad idea:

```txt
Allow 0.0.0.0/0 to port 5432
```

Better:

```txt
Allow only specific trusted IP addresses to port 5432
```

For example:

```txt
Only allow developer office IP
Only allow VPN IP
Only allow another server IP
```

Also require:

- SSL
    
- strong password
    
- restricted database user
    
- no superuser access
    
- backups
    
- logging
    
- monitoring
    

This option can work, but it needs discipline. Otherwise it becomes a future incident report wearing a fake mustache.

---

## Important rule: app access and human access are different

There are two different needs:

## 1. App-to-database access

This should usually happen privately.

Example:

```txt
Rails app container
  ↓
private Docker/Coolify network
  ↓
PostgreSQL container
```

The connection string may look like:

```txt
postgresql://user:password@postgres:5432/my_app_production
```

This is not meant for external laptops.

## 2. Human/developer database access

This should usually happen through:

```txt
SSH tunnel
VPN
managed DB dashboard
temporary restricted access
```

The mistake is trying to solve human access by making the production database public.

That is usually the wrong lever.

## Practical decision guide

When deciding how to expose a database, ask:

1. Is this database production?
    
2. Does it contain client or sensitive data?
    
3. Who needs access?
    
4. Is access temporary or permanent?
    
5. Can the app connect internally instead?
    
6. Can developers use SSH tunnel?
    
7. Can we use VPN?
    
8. Do we need a managed database provider?
    
9. If exposing publicly, can we restrict by IP?
    
10. Are SSL, backups, logs, and least-privilege users configured?
    

If I cannot confidently answer those questions, I should not expose the database publicly.

For a self-managed VPS with Coolify:

```txt
Public internet
  ↓
Only expose 80/443 for web traffic
  ↓
Coolify/reverse proxy routes apps by domain
  ↓
Apps connect to databases internally
  ↓
Database ports stay private
```

For developer access:

```txt
Use SSH tunnel or VPN
```

For easy external database URI access:

```txt
Use managed PostgreSQL
```

Avoid:

```txt
Opening PostgreSQL to the whole internet
```

unless there is a very specific reason and proper security controls.

## Final summary

The database URI was not the problem.

The real question was network exposure.

A PostgreSQL URI like this:

```txt
postgresql://user:password@host:5432/database
```

is just a set of connection instructions.

It still needs:

- an open port
    
- a reachable host
    
- PostgreSQL listening on that interface
    
- valid credentials
    
- proper firewall rules
    
- SSL/TLS if required
    
- secure user permissions
    

A website domain using HTTPS is not the same as a PostgreSQL database connection.

ElephantSQL can provide public database URIs because it is a managed database service designed for that.

On my own VPS, the safer default is:

```txt
Do not expose the database publicly.
Use internal networking for apps.
Use SSH tunnel or VPN for developer access.
Use managed DB if public URI access is required.
```

That is the lesson.

A URL is not security.

A domain is not a tunnel.

HTTPS is not PostgreSQL.

And opening a port is not “just generating a URL.” It is changing who can knock on the database door.
